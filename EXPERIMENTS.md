**Результат:** Loss 2.9470 на step 4800 (хуже V4 из-за 10 треков вместо 8).

### ROUTER-001: Профайлинг треков (50K примеров × 10 треков)

**Результат:** 🔴 **Базовый Track 0 побеждает в 93.2% примеров.**

**Что сделали:** Прогнали 50K примеров через каждый трек отдельно, считая cross-entropy loss. Для каждого примера определили лучший трек.

**Ключевое открытие:** Донорские треки (Gemma, Qwen, DeepSeek) **ХУЖЕ** базовых треков по ВСЕМ примерам. Причина: proj_in/proj_out проекции не обучились правильно. Они должны транслировать из нашего d=1024 в пространство донора и обратно, но это слишком сложная задача для линейных проекций при такой разнице размерностей.

**Вывод:** Подход "3 замороженных слоя + линейные проекции" — тупиковый. Нужен другой подход к интеграции знаний из больших моделей.

**Файлы:** `train_router.py`, `router/router_profile.pt`

### GATE-001: Обучение Gate (диспетчера)

**Что сделали:** На основе профайлинга обучили gate Linear(1024, 10) предсказывать лучший трек по входному embedding. Использовали KL divergence на soft targets из loss.

**Результат:** Gate работает, но бесполезен — предсказывает Track 0 для всех, потому что Track 0 действительно лучший для 93% примеров. Гейт верно отражает реальность, просто реальность плохая.

**Вывод:** Сначала нужно сделать треки реально разными (специализированными), потом учить gate. Gate — следствие, не причина.

### FEEDING-001: Кривая обучения — кормление узкой темой 🔥

**Дата:** 2026-04-08
**Гипотеза:** Можно ли научить маленькую предобученную модель узкой теме (грузинская кухня), просто скормив ей тексты?

**Сетап:**
- Базовая модель: `ai-forever/rugpt3small_based_on_gpt2` (125M params, уже говорит по-русски)
- Данные: 99 текстов про грузинскую кухню (~42K слов, 0.4MB)
- Обучение: lr=5e-5, batch=4, 300-500 steps на стадию
- GPU: RTX 3090 24GB, ~13.5 step/s
- Стадии кормления: [0, 5, 10, 25, 50, 99] примеров

**Результат — Кривая обучения:**

| Примеров | Perplexity | Изменение | Train Loss | Что происходит |
|----------|-----------|-----------|------------|----------------|
| 0 (база) | **121.4** | — | — | Знает русский, но тему знает поверхностно |
| 5 | 1929.1 | +1489% 📈 | 0.099 | Catastrophic forgetting! Зазубрил 5 текстов, сломал язык |
| 10 | 441.3 | +264% 📈 | 0.217 | Всё ещё хуже базы, язык ломается |
| 25 | 850.0 | +600% 📈 | 0.450 | Много новых фактов, модель путается |
| 50 | 649.9 | +435% 📈 | 0.685 | Начинает справляться, грузинская лексика появляется |
| **99** | **39.3** | **-67.6%** 🎉 | 0.583 | **ПРОРЫВ! Модель выучила тему** |

**Ключевое наблюдение — Catastrophic Forgetting при малых данных:**
При 5-50 примерах модель ХУЖЕ базы. Это catastrophic forgetting: 300 шагов на малом количестве текстов ломают общие языковые знания, а новых знаний недостаточно для компенсации. При 99 примерах достигается критическая масса данных — модель И выучивает тему И сохраняет язык.

**Аналогия:** Как попугай — если повторять 5 фраз 300 раз, он забудет остальные слова. Если сказать 99 разных фраз — он поймёт паттерн.

### FEEDING-003: Обратный тест — определение → угадай название 🔥

**Дата:** 2026-04-08
**Метод:** Даём описание блюда → надо назвать что это.

**Результат: Claude 10 : 1 Модель**

Модель НЕ УМЕЕТ НАЗЫВАТЬ вещи по описанию! Она обучена на текстах формата "Сулугуни — это рассольный сыр", поэтому может описать сулугуни, но не может угадать "рассольный сыр → сулугуни".

Единственное что угадала — сулугуни (видимо в данных был текст "рассольный сыр сулугуни").

**Критический вывод для продукта:** При подготовке обучающих данных нужны ДВУНАПРАВЛЕННЫЕ пары:
- "Ткемали — это кислый соус из алычи" (название → описание)
- "Кислый соус из алычи — это ткемали" (описание → название)
- "Вопрос: Как называется кислый соус из дикой сливы? Ответ: Ткемали" (Q&A формат)

Без этого модель умеет описывать, но не умеет определять/классифицировать.

**Вывод для системы уровней в Jippy:**
- 99 примеров (~42K слов) достаточно чтобы модель ПРИБЛИЗИЛАСЬ к уровню серверной модели (7:6 в прямом тесте). Но это не значит "обогнала" — нужен формальный бенчмарк с порогом.
- **Обязательное тестирование перед повышением уровня:** перед тем как объявить пользователю "твой Jippy обошёл серверную модель", нужно прогнать слепой бенчмарк (20+ вопросов, модель vs сервер, оба отвечают вслепую). Порог: модель должна стабильно побеждать (>60% вопросов).
- Это можно использовать для **предварительной оценки уровня**: по количеству скормленных данных (~100 примеров ≈ приближение к серверу, ~500+ ≈ уверенное превосходство), а затем доуточнять реальным бенчмарком.
- Пока бенчмарк не пройден — уровень показывается как "примерный" (прогресс-бар на основе объёма данных), после прохождения — подтверждённый.

## Ключевые находки и выводы

### 1. Broken Projections (корневая проблема V5)
Линейные проекции proj_in/proj_out между нашим пространством (d=1024) и донорским (d=3072-7168) не обучаются как часть frozen donor layers. Это как пытаться говорить на чужом языке через Google Translate — смысл теряется.

**Решение:** Дистилляция из полной модели с обучаемыми проекциями (DISTILL-003).

### 2. Sequence-level > Token-level дистилляция
Подача отдельных токенов убивает attention → плато 0.74. Полные последовательности → 0.90+.

### 3. Данные помогают, но логарифмически
5× данных = +0.02-0.03 cosine. Основной рост от архитектуры и шагов, не от объёма данных.

### 4. Screen vs nohup на Vast.ai
nohup процессы получают SIGHUP при disconnect SSH → умирают. Screen сессии выживают. Всегда использовать `screen -dmS`.

### 5. num_workers=0 обязателен при большой модели
DataLoader с num_workers>0 форкает процесс → копирует модель в RAM → OOM. При 20GB модели num_workers=0.

### 6. Кормление РАБОТАЕТ, но нужна критическая масса (~100 примеров)
5-50 примеров = catastrophic forgetting (модель ломает язык). 99 примеров (42K слов) = прорыв, perplexity падает в 3× лучше базы. Маленькая модель (125M) за 36 секунд обучения становится экспертом по узкой теме.

### 7. Модель НЕ умеет обратный вывод (описание → название)
Если обучать только на "X — это Y", модель описывает X, но не угадывает X по описанию Y. Нужны двунаправленные данные + Q&A пары.

### 8. При равных данных маленькая модель конкурирует с Claude на узкой теме
125M params модель набрала 7:6 против Claude по грузинской кухне (прямой тест). Ошибки — путает похожие блюда, что решается бОльшим объёмом данных (500-1000 текстов).

### 9. Прямая тренировка > дистилляция для узких доменов
Для супер-узких тем (грузинская кухня) — прямое обучение на текстах проще и эффективнее чем дистилляция из Gemma. Дистилляция лучше для широких доменов.

### 10. Два домена уживаются в одной модели (MOE-001)
Обучение на 1188 текстах (кухня + история) не сломало ни один домен. Грузинская кухня даже улучшилась (10/10 vs 18/20). Можно качать несколько веток навыков одновременно — как в Diablo.

### 11. Специалисты побеждают Claude в прямом тесте (MOE-002)
Два специалиста (125M params каждый) + keyword роутер набрали 5:3 против Claude в прямом тесте на 16 вопросах. 456-732 текста обучения, 2-3 минуты тренировки на RTX 3090.

### 12. Keyword роутер достаточен для PoC
Простой роутер на ключевых словах направил 19/20 вопросов верно. Для продакшена нужен нейронный роутер, но для PoC — keyword хватает.

## Планируемые эксперименты

### DISTILL-004: Узкая специализация трека (Python-only из Gemma)

**Гипотеза:** Если учить студента только на Python-коде, cosine будет значительно выше чем на всех данных. Узкий специалист > широкий генералист при том же размере.

**План:**
1. Отфильтровать v3_train.jsonl — только Python/код
2. Собрать активации Gemma только на этих данных
3. Обучить студента
4. Сравнить cosine с DISTILL-003

### DISTILL-005: Другие учителя

Повторить DISTILL-003 для:
- Qwen 3.5-27B (сильна в reasoning)
- GLM-5.1 (сильна в Chinese)
- Mistral-Small-24B (сильна в code)

Каждый учитель → свой специализированный трек.

### PERSONAL-001: Пользовательские треки

**Идея:** При первом знакомстве спрашиваем пользователя 3 интереса → доучиваем персональный трек (~50MB). Пользователь может докачивать знания на свой трек.

**Архитектура:**
- Базовые треки: предобученные специалисты (код, русский, English, math...)
- Персональный трек: инициализируется из ближайшего базового, fine-tune на данных пользователя
- Gate учится направлять к персональному треку для релевантных запросов

### ROUTING-001: Доменный роутинг

**Проблема:** Как определить "о чём" входящий текст и отправить нужному специалисту?

**Варианты:**
1. Простой classifier на embeddings (Linear → domain)
2. Keyword-based + fallback to neural
3. Hierarchical: сначала домен (code/text/math), потом конкретный трек
