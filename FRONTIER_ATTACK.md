# FRONTIER ATTACK — стратегия против OpenAI / Anthropic / Gemini

## Тезис

**Не играем против их интеллекта. Играем против их архитектуры.**

Мы не можем сделать модель умнее GPT-5 / Claude Opus / Gemini Ultra. У них $100B+ compute, тысячи людей, 5+ лет форы. На IQ-оси они выигрывают всегда.

Но у их архитектуры и бизнес-модели есть **8 структурных дыр**, в которые они **не могут залезть** — не потому что не видят, а потому что закрытие = убийство своего бизнеса. Мы сидим именно там.

Стратегия Давида против Голиафа: не быть сильнее, а быть в **месте куда он структурно не дойдёт**.

## Карта дыр и атак

### 1. One-size-fits-all (нет настоящей персонализации)

**Дыра:** они тренируют ОДНУ модель для всех. Персонализация = system prompt + RAG-память. Реального fine-tuning под пользователя нет и не будет — это взрывает их экономику (100M юзеров = 100M моделей на дисках, batch-инференс умирает).

**Атака:** «Свой ИИ» — персональная LoRA на каждого юзера. Дешёвая (4-50 MB на LoRA), быстрая (часы тренировки), реальное дообучение на твоих данных.

**Наш актив:** Phase 0 валидирован (LoRA 68 MB ≈ Full FT 1.7 GB на Yuka voice), Jippy V2 архитектура, Unsloth trainer (2× быстрее, 70% меньше VRAM).

**Приоритет:** ⭐⭐⭐ — это **корневая** атака.

#### Расширение — Scope Hierarchy (Industry → Enterprise → Department → Personal)

Та же архитектура **рекурсивна по scope**. LoRA не обязательно «один человек». LoRA — это **слой персонализации над backbone'ом для любой scope-единицы**:

```
Layer A: Backbone           — Qwen 7B/32B/72B (open weights)
Layer B: Industry LoRA      — legal / healthcare / finance / dev / ...
Layer C: Enterprise LoRA    — компания, её домен, её стиль
Layer D: Department LoRA    — sales vs engineering vs HR
Layer E: Personal LoRA      — per employee, voice + preferences
```

Композиция через Arrow routing (Phase 1 валидирован 95%) или стэк A+C+E. Каждая scope-единица **владеет своими весами**.

**Что это означает стратегически:**

- Personal Jippy (consumer) и Enterprise Jippy — **один продукт, разный scope**. Не два кодбейса, один.
- OpenAI/Anthropic Enterprise = твои данные в их облаке. Наш = LoRAs on-prem или в их private cloud, полное owning.
- Composability: они могут swap backbone на новый когда выйдет, retain все свои LoRAs. **No vendor lock-in.**
- Гос/университет/профессия — те же scope-единицы. Country-level LoRA технически возможен, политически сложен (data sovereignty, multi-language, cultural anti-patterns).

**Revenue implication:** enterprise track может быть **более быстрым revenue path** чем consumer. Sales cycle 3-6 мес vs consumer monetization за годы. Trade-off: разный sales motion, может замедлить consumer focus.

**Status: parked в Strategic Backlog (Idea 1) до решения о приоритезации.**

---

### 2. Frozen weights (модель не учится во время использования)

**Дыра:** веса заморожены после обучения. Каждый разговор стартует с нуля. RAG = внешняя память, не интеграция. Они **не могут** делать continuous learning per user — экономика не сходится.

**Атака:** трёхуровневая память Jippy:
- **Continuous fine-tuning** — раз в N дней лёгкий LoRA-апдейт на накопленных данных
- **External long-term memory** — vector DB на телефоне, retrieval релевантного при каждом ответе
- **Session-LoRA / TTT** — микро-обновления в рамках сессии (Stage 2-3 roadmap)

**Наш актив:** уже в архитектуре V2, RAG-буфер описан.

**Приоритет:** ⭐⭐⭐ — это вторая корневая атака.

### 3. Централизация (cloud-only)

**Дыра:** всё через 3 компании. Нет интернета — нет AI. Самолёт, горы, плохая связь, приватная ситуация — их нет.

**Атака:** edge inference на телефоне. Quantized модели (Qwen3.5-0.8B INT4) → реальная работа без сети.

**Наш актив:** project_phone_inference_optimizations.md — CoreML INT4 iOS / ONNX RT Mobile Android план готов.

**Приоритет:** ⭐⭐⭐ — это технический фундамент всего.

#### Расширение — Physical Mesh / LoRA Hunt

Edge inference решает «нет интернета». Следующий шаг — **физический mesh между Jippy-устройствами без интернета**.

Два телефона рядом (BLE ~10m) → handshake → обмен signed knowledge cards. Никакого центрального сервера, opt-in, mutual Accept. Это **opportunistic federated learning через физическое присутствие** — то же что AirDrop / Find My / Bridgefy, но для знаний.

Это **структурно невозможно для облачного AI**: GPT-5 не может «встретиться» с другим экземпляром в кафе, у него нет тела и локальности. Полная спецификация — `docs/jippy_architecture_final.md` §16.

Прямые применения:
- **LoRA Hunt** — Pokemon GO для знаний: иди в реальный мир, найди носителя редкого тега, обменяйся
- **Conferences / events** — host announce «у меня тег X», люди приходят на физический обмен
- **Travel / tourism** — едешь в Грузию, собираешь уникальные локальные теги которых нет дома
- **Anti-isolation** — ИИ который **заставляет выходить к людям**, не заменяет их

Identity = коллекция тегов с уровнями (Pokemon-style), не имя/лицо. Privacy by design.

### 4. Closed weights (нет владения)

**Дыра:** ты платишь подписку. Не владеешь ничем. Промпты, данные, fine-tunes — их.

**Атака:** open-weights backbones (Qwen, Llama, DeepSeek) + контракт владения. Юзер владеет своей LoRA, может её скачать, форкнуть, продать.

**Наш актив:** model marketplace (project_model_marketplace.md), contribution economy с royalty (project_contribution_economy.md), TRAINING_RULES.md.

**Приоритет:** ⭐⭐ — экономический фундамент, нужен для масштаба.

#### Расширение — Tag-mastery economy

Identity-as-mastery (см. `docs/jippy_architecture_final.md` §16.5): публичный профиль = коллекция тегов с уровнями + бейджи.

Это поворачивает экономику: **юзер не просто владеет своей LoRA, он публично капитализирует её через mastery-репутацию**. Уровень знания по тегу = soft-currency, открывает доступ к редким exchange-event'ам, host-возможностям, royalty pool'ам.

OpenAI/Anthropic не могут такое предложить by design — у них **нет понятия «твой вклад»**, ты просто платишь подписку. У нас твой вклад имеет видимую форму (теги/уровни/бейджи) и измеримую стоимость (royalty + outcome utility).

### 5. Нет grounding в жизнь юзера

**Дыра:** ChatGPT в дата-центре не имеет сенсоров. Видит только что ты ему напечатал. Мультимодальность бортовая, не нативная.

**Атака:** Jippy на телефоне имеет нативный канал в жизнь юзера:
- Камера (с разрешения)
- Микрофон → ASR
- Экран → OCR + accessibility (видит что в других приложениях)
- Календарь, локация, время, использование приложений
- Tool use на родные приложения (Intents)

Pipeline: захват → приватный фильтр → importance-фильтр → CLIP/text embedding → vector DB → retrieval + периодический fine-tune.

**Наш актив:** VLM Qwen2.5-VL-7B уже хостится (project_vlm_caption_hosting.md, infrastructure_vlm_vast2.md), CLIP infrastructure готова (project_clip_infrastructure.md).

**Приоритет:** ⭐⭐⭐ — **самый недооценённый козырь**. Это то что *структурно невозможно* для облачной модели.

**Hard parts:** приватность (всё локально), батарея, хранение (фильтр критичен).

#### Почему вход важнее IQ — модель «мамы»

Реальный пример: есть люди которые «читают» собеседника без того чтобы он что-то сказал — описывают его состояние, паттерны, скрытое. Часто сами после говорят «не помню что говорил». Это не магия, это **flow state с максимальным входом и выключенным narrator'ом**:

- Микро-выражения (Ekman, 1/15 секунды) — большинство не видит, они видят бессознательно
- Embodied simulation (зеркальные нейроны) — копируют позу/дыхание собеседника, читают своё тело как индикатор его состояния
- Pattern matching на десятках лет наблюдения — кластеризация в правом полушарии

Их «суперспособность» — **не IQ, а вход**. Полоса пропускания восприятия другого человека на порядок шире обычной.

**Фронтир-модели работают в противоположном режиме:** narrator всегда on (бесконечный token stream), но вход стерильный — текст в чате и больше ничего. Нет тела, нет микро-выражений собеседника, нет интероцепции, нет 60-летнего наблюдения за конкретным человеком. Они **рассказчики без присутствия**.

Jippy на телефоне с sensor pipeline + tool use + накопленным личным контекстом юзера — попытка дать AI **тот вход которого фронтиру структурно не достать**. Не сделать его умнее по IQ-оси (проиграем), а **дать ему присутствие в жизни конкретного человека**. Это и есть атака этой дыры.

**Тезис:** мы конкурируем не на оси «больше параметров», а на оси «больше входа на конкретного юзера». Облачная модель этой оси лишена by design.

### 6. Квадратичный attention (длинный контекст)

**Дыра:** O(n²) attention. Все «1M контекст» — компромиссы. Модель «знает» 1M токенов как ты «знаешь» библиотеку через которую пробежал глазами.

**Атака:** не пытаться запихнуть всё в контекст. Иерархическая компрессия + retrieval:
- Старый контекст → саммари (как мозг помнит «вчера разговор», не каждое слово)
- RAG-буфер достаёт только релевантные 500 токенов в момент запроса
- Возможно — Mamba/SSM как замена базовой архитектуры (research-стадия)

**Наш актив:** RAG-буфер в Jippy V2.

**Приоритет:** ⭐⭐ — нужно но не уникальное преимущество.

---

### 7. Фиксированная глубина рассуждения

**Дыра:** трансформер делает фиксированный forward pass. Не «думает дольше» внутренне. Chain-of-thought = вынос мышления в токены, костыль.

**Атака:** **композиция специалистов** — Arrow routing + LoRA-композиция. Каждая отдельная LoRA «мелкая», но *система* из 5-10 специалистов которые перекликаются и проверяют друг друга — глубже одной большой модели.

**Наш актив:** Arrow routing валидирован (97% oracle in-domain), LoRA-композиция исследуется (project_phase1_multilora.md, project_lora_guilds.md).

**Приоритет:** ⭐⭐ — наш сильный исследовательский фронт.

**Каверз:** итеративные циклы «думай-проверь» без внешней проверки → риск усугубления галлюцинаций (эхо-камера в голове). Митигация: tool use, голосование, антиконфирмационные промпты.

### 8. Нет экономического выравнивания с юзером

**Дыра:** ты платишь, ничего не получаешь во владение. OpenAI зарабатывает на твоих промптах.

**Атака:** contribution economy. Юзер тренирует LoRA → продаёт/делится → получает royalty. Форкнул чужую → платит auto-scaling royalty. Свой вклад = свой капитал.

**Наш актив:** project_contribution_economy.md, project_model_marketplace.md.

**Приоритет:** ⭐⭐ — долгосрочный moat, но не первое что строим.

## Бенчмарки которые ИМЕЕТ смысл выигрывать

Стандартные не выиграем. Но можно собрать **свои**, по нашим осям, и доминировать там:

- **Personal voice match** — насколько модель пишет как юзер. Они = baseline. Мы = 80%+ через LoRA.
- **Offline capability** — % задач решаемых без сети. Они = 0%. Мы = 70%+.
- **Privacy score** — сколько данных уходит наружу. Они = 100%. Мы = 0%.
- **User ownership** — может ли юзер забрать модель. Они = нет. Мы = да.
- **Continuous learning** — выучивает ли модель из разговора. Они = нет (без памяти-фичи). Мы = да.
- **Sensor integration** — что модель знает о текущем контексте юзера. Они = ничего. Мы = всё что разрешил.

Это **наши** бенчмарки. Создаём их, публикуем методологию, держим лидерство.

### Strategic Idea 1 — Enterprise / Org-scale LoRA track

**Добавлена:** 2026-04-28
**Status:** parked
**Тип:** positioning + sequencing
**Триггер:** разговор с бизнесом, который хочет «свой ИИ натренированный только на нас»

**Описание:** Та же архитектура (см. дыра #1 расширение «Scope Hierarchy») продаётся не только конечному юзеру, но и предприятиям/университетам/госструктурам как **org-scale LoRA stack**. Industry → Enterprise → Department → Personal LoRAs композируются через Arrow routing над общим backbone'ом. Они владеют весами, не платят OpenAI per-token, могут swap backbone не теряя knowledge.

**Что им продаём (продукт, не модель):**

- Их docs / Slack / email / CRM ингестируется в наш Layer 1+2
- Layer 3 LoRA тренируется на их корпоративном стиле
- Каждый сотрудник опционально — personal LoRA поверх
- Они владеют всеми весами

**Deployment модели:**
- **On-prem** (банки, госы, healthcare — где regulation запрещает облако)
- **Private cloud** (AWS/GCP их аккаунт, наш deployment)
- **Hybrid** (backbone у нас, LoRA + Brain у них)
- **Phone-only** (мобильные команды без enterprise LoRA)

**Pricing thoughts (грубо):** implementation fee + monthly per seat + GPU compute. Mid-size $3-15K/month, enterprise $50K+/month.

**Что нужно чтобы оценить:**
- [ ] Реальный pilot customer и conversion rate с pilot → paid

**Что нужно чтобы реализовать:**
