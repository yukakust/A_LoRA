# Jippy — Техническое задание (MVP)

> AI Тамагочи: корми, прокачивай, сражайся, зарабатывай.
> Маленькая AI-модель на телефоне пользователя учится узкой теме,
> растёт от кормления данными, сражается с серверными моделями-боссами,
> а после победы над финальным боссом — начинает зарабатывать.

---

## 1. Концепция продукта

Пользователь выращивает персонального AI-специалиста прямо на своём телефоне.

**Цикл:**
```
Выбрал тему → Кормишь данными → Модель учится на телефоне →
→ Задаёшь вопросы → Видишь как умнеет → Сражаешься с боссами →
→ Побеждаешь всех → Попадаешь в рейтинг → Помогаешь другим →
→ Зарабатываешь очки → Делишь реальные деньги
```

**Ключевая метафора:** Тамагочи + Diablo skill tree + Pokemon battles.

---

## 2. Технический стек

Стек идентичен `babylon-mvp`:

| Слой | Технология |
|------|-----------|
| Мобильное приложение | React Native 0.83 + Expo 55 + TypeScript |
| Веб-версия (опционально) | React 19 + Vite + TypeScript |
| Бэкенд | Node.js + Fastify 5 + TypeScript |
| База данных | PostgreSQL (Neon) |
| Локальная БД на телефоне | SQLite (react-native-sqlite-storage) |
| Хостинг бэкенда | Google Cloud Run (europe-central2) |
| Хостинг фронтенда | Firebase Hosting |
| CI/CD | GitHub Actions → Docker → Cloud Run |
| Push-уведомления | Firebase Cloud Messaging |
| Аутентификация | Firebase Auth (Google OAuth + Phone OTP) + JWT |
| Хранилище файлов | Firebase Storage |
| AI на телефоне | ONNX Runtime React Native (модель ~50MB) |
| AI на сервере (боссы) | Open-source модели через OpenAI-compatible API |

**Механика XP при кормлении:**
- Каждый новый факт/текст = XP, пропорционально новизне (perplexity)
- Повторная информация = мало XP ("Это я уже знаю! +1 XP")
- Новая информация = много XP ("Ого, это новое! +5 XP")
- В реальном времени: при наборе текста хлопушки "+1 XP" за новые слова по теме

**Управление ресурсами:**
- Обучение автоматически ставится на паузу при:
  - Уход с Wi-Fi (опционально, настраивается)
  - Батарея < 20%
  - Пользователь сворачивает приложение (продолжает в фоне, но с ограничением)
- Пользователь может вручную поставить паузу и продолжить позже
- Прогресс обучения сохраняется между сессиями

### 3.6.1. PvP — после финального босса ты сам становишься боссом

**После победы над Llama 70B:**
- Твои LoRA веса + knowledge_bank загружаются на сервер
- Ты появляешься в чужих аренах как "живой босс"
- Другие игроки могут вызвать тебя на бой

**Механика PvP "Я знаю, а ты?":**
```
Раунд 1: Jippy-A атакует
├── Берёт вопрос из СВОЕГО knowledge_bank
├── Jippy-B отвечает (инференс LoRA на сервере)
├── Сравниваем ответ B с эталоном A (keywords matching)
└── Score

Раунд 2: Jippy-B атакует (симметрично)
... 10 раундов (5 атак каждый)
```

**Что хранится на сервере:**
```
user_model/
├── base_model.onnx        # Общая 125M (одна копия)
├── lora_weights.bin        # ~2MB, персональный
└── knowledge_bank          # Q&A пары из кормления + keywords
```

**Оценка ответов (MVP):**
- Keyword matching: эталон содержит keywords, считаем совпадения
- Позже: LLM-судья (Phi-3) для более точной оценки

**Knowledge bank** формируется автоматически при кормлении:
- Каждый Q&A пар из pipeline → сохраняется в БД
- Keywords извлекаются из эталонного ответа

## 4. AI модель на телефоне

### 4.1. Архитектура модели

```
Модель Jippy (125M params, ~50MB на диске):
├── Embedding layer (vocab → d=1024)
├── 3 × Transformer layers (d=1024, 8 heads, ff_mult=4)
├── LM Head (d=1024 → vocab)
└── Tokenizer (SentencePiece 32K или GPT-2 BPE 50K)
```

- Формат на устройстве: **ONNX** (через `onnxruntime-react-native`)
- Инференс: ~5-10 ms/token на современных телефонах
- Размер на диске: ~50MB (INT8 квантизация)

### 4.2. Обучение на устройстве

**Ключевая техническая задача:** fine-tune модели прямо на телефоне.

Варианты реализации (в порядке приоритета):

**Результаты PoC-теста (Mac как прокси для iPhone):**

```
Модель: 125.2M params, 478 MB FP32, ~120 MB INT8
LoRA training: +71 MB RAM, 811 ms/step (Mac) → ~240 MB, ~2.4s (iPhone) ✅
Full training: +936 MB RAM, 5.7s/step (Mac) → ~1124 MB, ~17s (iPhone) ⚠️
Raw PyTorch inference: 221 ms/token (Mac) → СЛИШКОМ МЕДЛЕННО ❌
Нужен оптимизированный runtime (CoreML/ExecuTorch) для inference!
```

**Вариант A — CoreML (iOS) + ExecuTorch (Android) (ПРИОРИТЕТ):**
- Inference через CoreML / Apple Neural Engine — ожидаемо 10-50 ms/token
- LoRA training в PyTorch (фоновый thread) — 240 MB RAM, 2-3 сек/step
- PoC: конвертировать rugpt3small → CoreML, замерить inference speed
- Это единственный путь к быстрому inference на телефоне

**Вариант B — Гибрид (fallback если on-device training не получится):**
- Inference на телефоне через CoreML (быстро)
- Training на нашем GPU-сервере (5-30 секунд)
- Обновлённые LoRA веса скачиваются обратно (~2MB delta)
- Плюс: быстрее, надёжнее
- Минус: нужен интернет, серверные расходы

**Вариант C — Всё на сервере (MVP-minimum):**
- Если CoreML конвертация не получится в срок
- Модель + inference + training на сервере, клиент = UI

**Контроль ресурсов при обучении:**

### 4.3. Парсинг данных для кормления

| Источник | Реализация | Где обрабатывается |
|----------|-----------|-------------------|

| Текст | Прямой ввод | Телефон |

Когда Jippy победил финального босса и начинает помогать другим:
- Ответы проходят через автоматическую проверку на адекватность
- Фильтр мусора, спама, неправильной информации
- Автометрики: coherence score, factual consistency
- Жалобы от пользователей → ручная модерация

## 7. Серверная агрегация знаний ("Волосы → Вода")

### Концепция

Каждый пользователь тренирует узкого специалиста (волосы). По отдельности —
изолированные факты. Но когда сервер агрегирует знания тысяч экспертов,
возникают СВЯЗИ между доменами (вода). Кулинар + ботаник + историк = сервер
понимает ПОЧЕМУ грузинское вино делают в квеври (климат + глина + ферментация).

### Архитектура: Гибридная мета-модель

Мета-модель — не просто хранилище знаний и не просто роутер. Это **три в одном**:
база знаний + роутер + self-awareness (знает свои границы).

```
Телефоны (бесплатный inference):
├── 🐥 Jippy@user1 — грузинская кухня (125M, LoRA)
├── 🐥 Jippy@user2 — турецкая история (125M, LoRA)
├── 🐥 Jippy@user3 — виноделие (125M, LoRA)
└── ... × миллионы

        │ побил финального босса
        │ → верифицированные знания загружаются
        ▼

Сервер — ОДНА мета-модель (7→30B, растёт с пользователями):

  Вопрос пользователя
         ↓
    🌊 Мета-модель
    ├── Знает сама? → Отвечает (быстро, из объединённых знаний)
    ├── Не уверена? → Делегирует специалисту-LoRA домена
    └── Совсем не знает? → Честно: "Я этого пока не знаю"
```

**Три роли мета-модели:**
1. **База знаний** — содержит выжимку из всех доменов, видит кросс-доменные связи
2. **Роутер** — если вопрос слишком узкий, делегирует доменному LoRA-специалисту
3. **Честный арбитр** — знает границы своих знаний (см. секцию 7.1)

**Мета-модель работает как:**
- Финальный босс (самый сильный противник)
- Fallback когда Jippy на телефоне не знает ответ
- Кросс-доменный эксперт (связи между темами)
- Оркестратор для узких доменных LoRA

Связи между доменами не программируются — они возникают когда
одна модель видит все домены одновременно. Как GPT увидел триллион
токенов и научился связывать химию с кулинарией. Наше преимущество:
данные верифицированы экспертами (побили босса), а не мусор из интернета.

### Почему это бьёт GPT-5

| | GPT-5 | Jippy мета-модель |
|---|---|---|
| Источник знаний | Common Crawl (мусор + правда) | Только эксперты, прошедшие PvP |
| Галлюцинации | Уверенно выдумывает | "Я не знаю" (perplexity-порог) |
| Глубина в домене | Поверхностно обо всём | Экспертный уровень в каждом домене |
| Обновление знаний | Раз в N месяцев | Каждый день (от новых экспертов) |
| Размер | Триллион параметров | 7-30B, но с правильными данными |

Мы не строим "лучшую модель по бенчмаркам". Мы строим модель, которая
**знает правду лучше всех** — потому что её учили проверенные люди,
а не скрейпер интернета.

### 7.1. Механизм честности (Calibrated Uncertainty)

**Ключевой принцип: модель САМА говорит нам, когда она не уверена.**
Нам не нужно учить её говорить "я не знаю" — нужно слушать её собственный
сигнал уверенности.

**Как это работает:**

Perplexity (PPL) — мера "удивления" модели. Доказано экспериментально:
- Знакомый текст → PPL низкий (модель уверена, ответ надёжный)
- Незнакомый текст → PPL высокий (модель гадает, ответ ненадёжный)

```
Пользователь: "Какой соус к хинкали?"
  → Модель генерирует: "Ткемали из алычи"
  → PPL токенов ответа: 3.2 (низкий)
  → Уверенность: ВЫСОКАЯ ✅ → Отдаём ответ

Пользователь: "Какая столица Бурунди?"
  → Модель генерирует: "Бужумбура"
  → PPL токенов ответа: 89.4 (высокий)
  → Уверенность: НИЗКАЯ ⚠️ → "Я не уверен в этом ответе"

Пользователь: "Расскажи про квантовую физику"
  → Модель генерирует: бессвязный текст
  → PPL токенов ответа: 200+ (очень высокий)
  → Уверенность: НЕТ ❌ → "Это вне моей специализации"
```

**Три уровня честности:**

**Уровень 1 — Чистые данные (фундамент):**
Модель не может врать о том, чему её не учили врать. Данные поступают
только от экспертов, прошедших PvP-верификацию → мусора в обучении нет →
нечего воспроизводить как мусор.

**Уровень 2 — Runtime-проверка через perplexity (автоматическая):**
```
Генерация ответа → Измерение PPL токенов ответа
  ├── PPL < T1 (порог_уверен)     → ✅ Выдаём ответ
  ├── T1 < PPL < T2 (порог_нет)   → ⚠️ "Думаю что X, но не уверен"
  └── PPL > T2                     → ❌ "Я этого не знаю"
```
Пороги T1 и T2 калибруются автоматически: прогоняем тест на данных,
на которых модель тренировалась, и находим точку, где начинаются
галлюцинации. Это наш порог.

**Уровень 3 — Обучение границам (явное):**
В тренировочные данные добавляем негативные примеры:
```jsonl
{"text": "Q: Какой соус к хинкали? A: Ткемали — кислый соус из алычи"}
{"text": "Q: Какая столица Бурунди? A: Этот вопрос выходит за рамки моей специализации"}
{"text": "Q: Расскажи про квантовую физику A: Я специализируюсь на грузинской кухне"}
```

**Почему это работает у нас, но не у OpenAI:**

GPT тренирован на ВСЁМ интернете — у него нет чёткой границы "знаю / не знаю".
Его perplexity низкий даже на фейках, потому что он видел их в трейнинге.

У Jippy граница **кристально чёткая**: модель видела только проверенные данные
по конкретным доменам. Всё за пределами — территория высокого PPL = территория
честного "я не знаю".

### 7.2. Персональный когнитивный фреймворк ("Думает как ты")

**Проблема:** ChatGPT думает одинаково для всех. Два маркетолога с разными
стратегиями получают один и тот же ответ. Jippy должен думать как КОНКРЕТНЫЙ
пользователь — в его парадигме, с его приоритетами.

**Ключевой инсайт:** данные которыми пользователь кормит Jippy — это не просто
факты. Это его ФРЕЙМВОРК МЫШЛЕНИЯ. Если кормить статьями про "retention > acquisition"
и "community-led growth" — модель усваивает не только факты, но и приоритеты.

**Проблема реализации:** 125M модель на телефоне слишком маленькая чтобы
"переформулировать" ответ серверной модели через свою призму. Она не умеет
синтезировать — только запоминать и оценивать.

**Решение — три уровня, каждый усиливает предыдущий:**

**Уровень 1 — RAG с данными пользователя (MVP):**
```
Вопрос: "Как выйти на рынок Японии?"

1. Jippy на телефоне: PPL высокий → "не знаю" → идём на сервер

2. Сервер: vector search по базе кормления пользователя,
   достаём его ключевые тексты/приоритеты:
   - "retention > acquisition"
   - "community-led growth"
   - "organic > paid"

3. Мета-модель получает промпт:
   "Ответь на вопрос, учитывая что пользователь
    считает важным: [его тексты как контекст]"

4. Ответ: "Для Японии стройте community в LINE,
   retention-first подход, не лейте paid ads..."
```
Технология: обычный RAG с персонализацией. Проверено, надёжно, работает сегодня.

**Уровень 2 — PPL-ранжирование на телефоне (V1):**
```
1. Сервер генерирует 3-5 вариантов ответа:
   A: "Запустите paid ads в LINE..."          → PPL = 47 (чуждо)
   B: "Постройте community в LINE, retention..." → PPL = 12 (близко!) ✅
   C: "Наймите местное PR-агентство..."       → PPL = 35 (нейтрально)

2. Jippy (125M) на телефоне считает PPL каждого варианта
3. Выбирает вариант с минимальным PPL = ближайший к фреймворку
```
Модель не "думает" и не "интерпретирует" — она ОЦЕНИВАЕТ. PPL как детектор
совместимости с мировоззрением пользователя. 125M отлично справится с оценкой —
ей не нужно генерировать, только считать вероятности. Быстро и дёшево.

**Уровень 3 — Персональный LoRA на серверной модели (V2):**
```
1. Данные кормления пользователя хранятся на сервере
2. Сервер тренирует LoRA-адаптер на мета-модели (30B)
   используя данные ЭТОГО пользователя (~10MB LoRA)
3. При запросе: мета-модель + LoRA пользователя =
   большая модель, которая БУКВАЛЬНО думает как пользователь
```
vLLM поддерживает multi-LoRA serving (подгрузка адаптера per-request).
Это самый мощный уровень: 30B модель с кросс-доменными знаниями,
но с ТВОИМИ приоритетами на уровне весов, а не промпта.

**Roadmap персонализации:**
```
MVP:  RAG (данные пользователя как контекст → мета-модель)
V1:   + PPL-ранжирование (телефон выбирает ближайший ответ)
V2:   + Персональный LoRA на сервере (модель думает как ты)
```

Каждый уровень усиливает предыдущий, не заменяет.

### Фильтр качества (встроенный)

На сервер попадают знания ТОЛЬКО от тех, кто побил финального босса.
Это сертифицированные эксперты — мусор отсеивается автоматически.

### Цикл улучшения

```
Пользователи кормят Jippy (волосы)
    ↓
Побеждают босса = верифицированные знания
    ↓
LoRA дельты агрегируются на сервере (вода)
    ↓
Мета-модель умнеет + лучше знает свои границы
    ↓
Боссы становятся сильнее
    ↓
Нужно кормить ещё больше чтобы побить
    ↓
Ещё больше качественных знаний → 🔄
```

### Масштабирование инфраструктуры

| Фаза | Пользователи | Серверное железо | ~Стоимость |
|------|-------------|-----------------|-----------|
| MVP | <1K | 1× A100 (4 босса) | ~$200/мес |
| Рост | 1K–50K | 2-4× A100, еженедельная агрегация | ~$1K/мес |
| Масштаб | 50K–500K | 8-16× A100, мета-модель 7B | ~$5K/мес |
| Большой | 500K–5M | 32-64× GPU, мета-модель 15-30B | ~$20K/мес |
| Огромный | 5M+ | Кластер, мета-модель 30B+ | ~$50K/мес |

**Почему дёшево:**
- Тяжёлый inference (чат) — на телефонах БЕСПЛАТНО
- Сервер только для batch-тренировки (по ночам) + боссы (редкие события)
- Одна мета-модель, не тысяча отдельных моделей
- Расходы растут с доходом (больше экспертов = модель ценнее = больше платят)

**Агрегация:** не сырые данные пользователей, а LoRA дельты (~2MB).
1000 LoRA → Federated Averaging → одна улучшенная мета-модель.
Тренировка на агрегированных знаниях = часы на одном GPU.

## 10. Открытые вопросы для исследования

1. **On-device training:** Насколько реально ONNX Training на React Native? Нужен PoC в первую неделю. Если нереально — переходим на гибридный вариант (обучение на сервере).

2. **Размер модели vs качество:** 125M (50MB) — достаточно ли для телефона? Или нужна модель поменьше (30M, ~15MB)? Нужен бенчмарк на реальных устройствах.

3. **Судейство финального босса:** Как организовать оценку реальными людьми? Формат: случайные эксперты из рейтинга оценивают A vs B обезличенно? Сколько оценок нужно для надёжности?

4. **GPU для боссов:** Llama 70B требует ~40GB VRAM. Хостить на нашем сервере или использовать API-провайдер (Together/Groq)?

5. **Антимусор после финального босса:** Как валидировать качество данных? Автоматический fact-checking? Или достаточно оценок от других экспертов?
